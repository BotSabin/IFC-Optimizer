# IFC Optimizer Pro Web

Professional BIM web platform for opening, visualizing, analyzing, optimizing, and exporting very large IFC models.

## Stack

- Frontend: React, TypeScript, Vite, TailwindCSS, Three.js, That Open Engine dependency hooks.
- Backend: FastAPI, SQLAlchemy, Celery, Redis, PostgreSQL.
- IFC processing: IfcOpenShell-ready service layer with deterministic development fallback.
- Storage: local development storage with S3-compatible settings prepared for production.

## Features Included

- Drag/drop IFC upload UI with size and validation/progress states.
- Analyzer metrics for entities, products, classes, geometry, triangles, properties, and quantities.
- Sortable-style class statistics data model and dense BIM class tree.
- Hide, show, isolate, select, Ctrl-click, and Shift-click class interactions.
- Element browser with search, filter, sort, and double-click zoom/flash behavior.
- Three.js viewer shell with orbit/section/measure/navigation toolbar controls and That Open Engine integration points.
- Native color and class color override model.
- Safe, medium, and aggressive optimization estimates.
- Background task APIs for analysis, optimization, class deletion, IFC export, GLB export, and geometry cache generation.
- Docker Compose stack for API, worker, PostgreSQL, Redis, and frontend.

## Run With Docker

```bash
docker compose up --build
```

Open:

- Frontend: http://localhost:5173
- API docs: http://localhost:8000/docs

## Run Locally

Quick start on this Mac after the dependency setup already done by Codex:

```bash
./scripts/start-local.sh
```

Backend:

```bash
cd backend
python -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
uvicorn app.main:app --reload
```

Worker:

```bash
cd backend
celery -A app.core.celery_app.celery_app worker --loglevel=info
```

Frontend:

```bash
cd frontend
npm install
npm run dev
```

For this workspace, a local Node.js runtime was installed under `work/runtime` one level above the deliverable folder. If your shell does not have Node/npm globally, either install Node.js normally or run npm with that local runtime on this Mac.

## GitHub Pages

The frontend can be published to GitHub Pages using `.github/workflows/deploy-frontend.yml`.

1. Push this folder as a GitHub repository.
2. In GitHub, enable Pages with source set to GitHub Actions.
3. Add a repository variable named `VITE_API_BASE_URL`.
4. Set it to the public backend URL, for example `https://ifc-optimizer-api.example.com`.

GitHub Pages only hosts the browser app. IFC upload, IfcOpenShell analysis, optimization, exports, and background jobs still require the FastAPI backend to run on a server, VM, Render, Railway, Fly.io, AWS, Azure, or a company-accessible machine.

## Netlify + Backend

Netlify is a good fit for the frontend only. This repo includes `netlify.toml`, so Netlify can build from the `frontend` folder automatically.

Set this Netlify environment variable:

```text
VITE_API_BASE_URL=https://your-backend-url
```

For the backend, use a service that supports long-running Python processes and workers. Render is prepared through `render.yaml` and includes:

- FastAPI web service
- Celery worker
- PostgreSQL
- Redis

After Render creates the backend URL, add the frontend URL to `IFC_CORS_ORIGINS` on Render and set the same backend URL as `VITE_API_BASE_URL` on Netlify or GitHub Pages.

## IfcOpenShell Integration

The backend service attempts to import `ifcopenshell` during analysis. If unavailable, it returns deterministic development data so the platform stays usable while infrastructure is being prepared. Install IfcOpenShell in the backend image or environment to enable real IFC parsing.

## That Open Engine Integration

The frontend includes `@thatopen/components`, `@thatopen/components-front`, and `three` dependencies. Replace the placeholder viewer scene in `frontend/src/components/Viewer.tsx` with a production world/components setup, then stream geometry cache chunks generated by backend workers.
